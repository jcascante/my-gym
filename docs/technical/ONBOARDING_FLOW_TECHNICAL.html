⚙️ Conditional Onboarding Flow Technical Documentation

Architecture and implementation details for the first-time user onboarding system

Overview Architecture API Endpoints Data Models Data Flow Implementation Testing Deployment

Overview

The conditional onboarding flow ensures new users complete their profile before accessing the dashboard, while returning users skip directly to the app. This improves UX by showing relevant content to each user type without forcing re-onboarding on every login.

Key principle: First login shows onboarding form (profile is null) → Returning login skips to dashboard (profile exists).

The onboarding form itself (OnboardingPage.tsx) is a client-only, 6-step wizard — one <form> element that reveals a different field group per step via conditional rendering, rather than six separate routes or forms. The core profile submission is unchanged: the underlying POST /api/v1/users/profile contract and payload still submit one combined profileData object on the final step. The one exception is the "Training Environments" step (see Training Environments Technical Reference), which persists each environment immediately via its own POST /api/v1/training-environments call as it's added, rather than batching it into the final profile submission — environments are an independent resource, not part of UserProfile.

Architecture

System Components

Frontend (React + Zustand) ↓ Auth Store (user + userProfile) ↓ App Routing (if !userProfile → OnboardingPage else DashboardPage) ↓ API Client (auth.ts) ↓ Backend (FastAPI) ↓ User Service (CRUD operations) ↓ Database (PostgreSQL)

State Management

The auth store in Zustand maintains two key pieces of state:

  • user: Current authenticated user (email, id, name)
  • userProfile: User's fitness profile (goals, experience, availability) or null

On login, the frontend fetches the current user and their profile. If profile is null, user sees OnboardingPage. If profile exists, user goes to DashboardPage.

API Endpoints

Get Current User with Profile

GET /api/v1/users/me

Purpose: Retrieve current authenticated user with their profile (if it exists)

Auth: Requires valid JWT token

Response (200):

{ "id": 1, "email": "user@example.com", "name": "John Doe", "userProfile": { "userId": 1, "fitnessGoal": "build_muscle", "experience": "intermediate", "hoursPerWeek": 4, "createdAt": "2026-07-09T10:00:00Z" } // null if no profile yet }

Create or Update User Profile

POST /api/v1/users/profile

Purpose: Create a new profile or update existing one (partial updates preserve fields)

Auth: Requires valid JWT token

Request:

{ "fitnessGoal": "build_muscle", "experience": "beginner", "hoursPerWeek": 3 }

Response (201 on create, 200 on update):

{ "id": 1, "userId": 1, "fitnessGoal": "build_muscle", "experience": "beginner", "hoursPerWeek": 3, "createdAt": "2026-07-09T10:00:00Z", "updatedAt": "2026-07-09T10:00:00Z" }

Data Flow

First-Time User Flow

1. User signs up 2. Backend creates User record (no profile yet) 3. User logs in → /login 4. Frontend POST /auth/login with credentials 5. Backend returns {accessToken, refreshToken, user} 6. Frontend fetches GET /users/me 7. Backend returns {user, userProfile: null} 8. Frontend setAuth(user, null) in store 9. App routing detects !userProfile → shows OnboardingPage 10. User fills form (goal, experience, hours) 11. Frontend POST /users/profile with form data 12. Backend creates UserProfile record 13. Frontend receives profile response 14. Frontend setUserProfile(profile) 15. App routing detects userProfile exists → shows DashboardPage

Returning User Flow

1. User logs in → /login 2. Frontend POST /auth/login with credentials 3. Backend returns {accessToken, refreshToken, user} 4. Frontend fetches GET /users/me 5. Backend returns {user, userProfile: {...}} 6. Frontend setAuth(user, profile) in store 7. App routing detects userProfile exists → shows DashboardPage 8. User is immediately in app (no onboarding)

Profile Update Flow

1. User in app, clicks Settings → Edit Profile 2. Form pre-populates with existing profile data 3. User modifies field(s) 4. Frontend POST /users/profile with partial data 5. Backend merges with existing profile (preserves unmodified fields) 6. Backend returns updated profile 7. Frontend updates store with new profile 8. UI reflects changes immediately

Frontend: Multi-Step Wizard

File: frontend/src/pages/OnboardingPage.tsx

A STEPS constant defines each step's title and which fields are required before the user can advance:

const STEPS: { title: string; requiredFields: (keyof OnboardingFormData)[] }[] = [ { title: 'Personal Information', requiredFields: ['age', 'gender', 'weight_kg', 'height_cm'] }, { title: 'Fitness Level', requiredFields: ['experience_level', 'activity_level'] }, { title: 'Workout Preferences', requiredFields: ['fitness_focus', 'days_per_week', 'workout_duration_min'] }, { title: 'Training Environments', requiredFields: [] }, // optional — see below { title: 'Your Goals', requiredFields: [] }, { title: 'Additional Information', requiredFields: [] }, ];

All form fields live in one formData state object for the lifetime of the page — steps only control which subset is rendered ({currentStep === 0 && (...)}), so navigating Back and Next never loses previously entered values. equipment was removed from OnboardingFormData and the Workout Preferences step entirely — it moved to the per-environment equipment_tags field described below.

The Optional Training Environments Step

Step index 3 renders zero or more read-only TrainingEnvironmentCards for environments already added this session, plus either an "Add Environment" button or an inline TrainingEnvironmentForm. Unlike the rest of the wizard, this step's data isn't held in formData — each environment is persisted immediately on save via createTrainingEnvironment() (POST /api/v1/training-environments), and the returned record is appended to local environments state purely for that step's display. STEPS[3].requiredFields is [], so handleNext never blocks on it — zero environments is a valid, supported state.

const handleAddEnvironment = async (payload: TrainingEnvironmentPayload) => { setEnvironmentError(null); try { const environment = await createTrainingEnvironment(payload); setEnvironments((prev) => [...prev, environment]); setShowAddEnvironment(false); } catch (err) { setEnvironmentError(getErrorMessage(err)); } };
⚠️ UX gotcha: two independent sets of Save/Cancel and Back/Next controls

The inline TrainingEnvironmentForm has its own Save/Cancel buttons, separate from the wizard's own Back/Next. Early user testing showed that clicking the wizard's Next while the form was still open silently discarded whatever had been typed, since that data only becomes a saved TrainingEnvironment via the form's own submit handler. Fixed by disabling both wizard Back and Next buttons whenever showAddEnvironment is true, and by showing an inline hint pointing at the "Save Environment" button:

<Button type="button" ... onClick={handleNext} disabled={showAddEnvironment}> Next </Button>

See Training Environments Technical Reference for the full TrainingEnvironment model, CRUD API, and the ProgramCreationRequest placeholder that follows once a user has at least one saved environment.

handleNext checks the current step's requiredFields against formData before advancing; if any are empty it sets a form error and stays on the same step instead of calling setCurrentStep:

const handleNext = () => { const missingFields = STEPS[currentStep].requiredFields.filter((field) => !formData[field]); if (missingFields.length > 0) { setError('Please fill in all required fields before continuing.'); return; } setError(null); setCurrentStep((step) => step + 1); };

handleSubmit is unchanged except for a guard that no-ops unless currentStep is the last step — this means the browser's native Enter-to-submit only fires the real API call from the final step, since only that step renders a type="submit" button. A progress bar reflects (currentStep + 1) / STEPS.length as a percentage width.

Note: Field-presence validation (requiredFields) is client-side UX only. The backend's UserProfileRequest schema still defines the authoritative required/optional fields and rejects invalid payloads independently of what the wizard allowed through.

Deployment Considerations

Database Migrations

New UserProfile table requires migration:

docker-compose exec backend alembic upgrade head

Environment Variables

No new environment variables required. Existing DATABASE_URL and JWT_SECRET are sufficient.

API Versioning

New endpoints are under /api/v1/ (same version as existing auth endpoints), ensuring backward compatibility.

Error Handling

  • 401 Unauthorized: Missing or invalid JWT token
  • 400 Bad Request: Invalid profile data
  • 404 Not Found: User not found (shouldn't happen for authenticated requests)

Monitoring

Key metrics to track:

  • Time from login to dashboard (should be instant for returning users)
  • Time to complete onboarding (should be <5 min)
  • Profile creation success rate
  • GET /users/me latency (should be <100ms)

Rollback Plan

If issues arise:

  1. Disable profile fetch on login (comment out getCurrentUser in LoginPage)
  2. Always show DashboardPage (bypass routing logic)
  3. Rollback database migration
  4. Deploy updated backend
