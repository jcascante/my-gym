Overview

MyGym implements JWT-based authentication with secure password hashing to provide user account management and session handling. Users can sign up with email/password, receive JWT tokens for subsequent requests, and log out to end their session. The system uses bcrypt for password hashing with SHA-256 pre-hashing for passwords exceeding bcrypt's 72-byte limit.

Key Features:

  • User registration with email validation
  • Secure password hashing (SHA-256 + bcrypt)
  • JWT access and refresh tokens
  • HTTP-only cookie storage
  • Async/await database operations
  • Protected routes with dependency injection

Security Mechanisms

Component Description
Password Hashing SHA-256 pre-hashing for passwords > 72 bytes, then bcrypt with 12 rounds
JWT Tokens Access tokens (15 min) + Refresh tokens (7 days)
HTTP-Only Cookies Secure token storage, prevents XSS attacks
CORS Cross-origin requests validated against allowed origins
Token Validation All protected routes require valid, non-expired JWT

API Endpoints

Sign-Up

Create a new user account.

POST /api/v1/auth/signup
Content-Type: application/json { "email": "user@example.com", "password": "secure_password_123", "first_name": "John", "last_name": "Doe" }
Response (201 Created):
{ "access_token": "eyJ0eXAiOiJKV1QiLCJhbGc...", "refresh_token": "eyJ0eXAiOiJKV1QiLCJhbGc...", "token_type": "bearer" }
Validation:
  • email: Required, valid email format, unique
  • password: Required, minimum 8 characters
  • first_name: Required
  • last_name: Optional
Error Responses:
  • 400 Bad Request: Invalid email, password too short, email exists
  • 422 Unprocessable Entity: Missing required fields
  • 500 Internal Server Error: Database error

Login

Authenticate and receive tokens.

POST /api/v1/auth/login
Content-Type: application/json { "email": "user@example.com", "password": "secure_password_123" }
Response (200 OK):
{ "access_token": "eyJ0eXAiOiJKV1QiLCJhbGc...", "refresh_token": "eyJ0eXAiOiJKV1QiLCJhbGc...", "token_type": "bearer" }
Error Responses:
  • 401 Unauthorized: Invalid email or password
  • 422 Unprocessable Entity: Missing required fields

Logout

End user's session.

POST /api/v1/auth/logout
Response (200 OK): { "message": "Successfully logged out" }
Note: Logout signals frontend to clear tokens. Tokens expire naturally via JWT exp claim.

Implementation Details

Password Hashing

Location: backend/app/core/security.py:13-16

def hash_password(password: str) -> str: if len(password.encode()) > 72: password = hashlib.sha256(password.encode()).hexdigest() return pwd_context.hash(password)

Why SHA-256 pre-hashing?

  • Bcrypt has a 72-byte password limit
  • Converting to 64-char hex string ensures compatibility
  • No security loss: bcrypt still provides strong hashing
  • Backward compatible with systems enforcing 72-byte limit

Data Flow

Sign-Up Flow

  1. User Action: User fills sign-up form and clicks "Sign Up"
  2. Frontend Validation: Check email format, password length, password match
  3. API Request: POST /api/v1/auth/signup with email, password, name
  4. Backend Processing:
    • Validate email is unique
    • Hash password (SHA-256 + bcrypt)
    • Insert User record into database
    • Generate access + refresh tokens
    • Set HTTP-only cookies
  5. Frontend Update: Store tokens, redirect to onboarding

Login Flow

  1. User Action: User fills login form and clicks "Log In"
  2. Frontend Validation: Check email format, password not empty
  3. API Request: POST /api/v1/auth/login with email, password
  4. Backend Processing:
    • Query database for user by email
    • Verify password matches
    • Generate tokens
    • Set HTTP-only cookies
  5. Frontend Update: Store tokens, redirect to dashboard

Deployment Considerations

Environment Variables

Production requires these environment variables:

SECRET_KEY="<strong-random-string-min-32-chars>" DATABASE_URL="postgresql://user:pass@host:5432/db" CORS_ORIGINS='["https://yourdomain.com"]' ACCESS_TOKEN_EXPIRE_MINUTES=15 REFRESH_TOKEN_EXPIRE_DAYS=7
Generate SECRET_KEY:
python -c "import secrets; print(secrets.token_urlsafe(32))"

Security Headers

Enable in production ASGI middleware:

  • Set cookie flags: secure=True, httponly=True, samesite="Strict"
  • Add headers: Strict-Transport-Security, X-Content-Type-Options, X-Frame-Options

Rate Limiting

Protect against brute force:

  • Limit login attempts: 5 per minute per IP
  • Implement CAPTCHA after 3 failed attempts
  • Log suspicious patterns

Troubleshooting

❌ "Password cannot be longer than 72 bytes"

Cause: Old password hashing without SHA-256 pre-hashing

Fix: Update to latest security.py with pre-hashing

Location: backend/app/core/security.py:14-15

❌ "Invalid credentials" with correct password

Cause: Password hashing mismatch or bcrypt version incompatibility

Fix:

  • Check pyproject.toml has bcrypt<5.0.0
  • Verify both hash and verify functions have same pre-hashing logic
  • User may need password reset if stored with old algorithm
❌ Tokens not persisted across requests

Cause: Missing credentials: 'include' or CORS issues

Fix:

  • Check fetch() includes credentials: 'include'
  • Verify CORS allows credentials
  • Check cookies set with httponly=True
❌ User not authenticated after login

Cause: Token expired or JWT secret changed

Fix:

  • Check token not expired
  • Verify SECRET_KEY is consistent
  • Ensure database session active in get_current_user()
