1. Overview

MyGym's program-generation engine is a pure rules-based system that takes a user's profile and preferences, then produces a structured workout program through a deterministic pipeline. The engine is:

  • Fast: Sub-100 ms generation, comfortably under the 2-second target
  • Deterministic: No randomness; identical inputs always produce identical outputs
  • Co-designed: Users refine programs through a match → draft → feedback → accept loop
  • LLM-ready: A pluggable seam for natural-language feedback parsing without touching the core engine

Architecture: The engine stores the program's base week once (sets, reps, movement patterns), then derives each week's concrete loads and reps at read time by applying progression models to the base scheme and week number. This keeps re-adaptation fast and matches the "lightweight engine" philosophy.

2. Layered Template Model

A ProgramTemplate is composed of three independent, swappable layers:

  • Split: The weekly session structure (e.g., Upper/Lower ×4). Each session is an ordered list of slots.
  • Slot rules: Each slot describes what fills it (movement pattern or body region, muscle emphasis, priority, set/rep scheme), not a specific exercise.
  • Progression reference: A {model_key, params} pair pointing at a code-defined progression model.

Example (Upper/Lower ×4):

split: sessions: - key: upper_a name: "Upper A" order: 1 slots: - { pattern: horizontal_push, priority: primary, scheme: main } - { pattern: horizontal_pull, priority: primary, scheme: main } - { pattern: vertical_push, priority: secondary, scheme: accessory } - { region: core, priority: accessory, scheme: accessory } progression: model_key: linear_load params: { increment_source: environment }

scheme references named set/rep schemes (e.g., main = 3×5, accessory = 3×10) that the progression model consumes to produce concrete sets, reps, and load for each week.

3. Data Model

Four new database tables underpin the engine:

Table Purpose Key Fields
program_templates The catalog of blueprints id, name, slug, goals[], experience_levels[], days_per_week_min/max, split (JSON), progression_ref (JSON), is_active
workout_programs A generated program instance id, user_id, template_id, status, duration_weeks, start_date, constraints (JSON)
workouts One row per session in the program's base week id, program_id, key, name, focus, order
workout_exercises One row per slot in a session id, workout_id, exercise_id, sets, reps_min, reps_max, base_load, is_locked, is_user_swapped

Program status: draft → active → archived

Constraints JSON example:

{ "locked_slots": [12, 15], "excluded_exercise_ids": [88], "swaps": { "14": 42 }, "volume_adjustments": { "upper_a": -1, "program": 0 }, "required_inputs": { "bench_press_start": 60, "squat_start": 80 } }

The constraints overlay accumulates user edits and is the single source of truth for re-adaptation. When a user makes a change, the engine re-runs affected slots while preserving everything locked.

4. Progression Models

Progression models define how weight and reps evolve week-to-week. All four are implemented in code, referenced by a model_key:

Model Behavior
linear_load Fixed increment added each week at fixed sets/reps. Increment sourced from the environment's available_weight_increments.
double_progression Work a rep range; add reps to the top across all sets, then add load and reset to the bottom.
weekly_undulating Vary intensity/reps across the week (heavy/light/volume days) for the same lift.
deload (modifier) Wraps any base model; every Nth week cuts volume/intensity, then resumes the climb.

Models implement a pluggable protocol and are registered in a keyed registry resolved from the template's model_key. Deload is a decorator that wraps a base model.

5. Matching Algorithm

The engine scores each template against the user's profile as a weighted sum:

Factor Weight Rule
Goal match High Template goals contains profile fitness_focus
Experience match High Template experience_levels contains profile experience_level
Days/week fit Medium Requested days_per_week within template range; distance-penalized
Session duration fit Medium Requested duration within template range; distance-penalized
Equipment feasibility Gate A template whose required patterns cannot be filled by the environment is excluded

Returns the ranked top 3 templates with a fit % and per-factor breakdown. Deterministic, sub-millisecond.

5.1 Scoring Depth: Matching Formula & Factors

The matching algorithm uses a weighted cumulative score to rank templates. Each factor contributes a normalized score (0–100) multiplied by its weight, then summed.

Matching Factor Breakdown

Factor Weight Calculation Notes
Goal Match 25% 100 if template.goals contains profile.fitness_focus else 0 Primary gate; heavy weight ensures templates match user's primary goal
Experience Match 25% 100 if template.experience_levels contains profile.experience_level else 0 Safety check; beginner users won't get advanced techniques
Days/Week Fit 20% 100 - (50 * abs_distance / max_distance) Distance-penalized; requesting 5 days fits a 4–6 range better than a 3-day template
Session Duration Fit 15% 100 - (50 * abs_distance_minutes / 30) Soft penalty for duration mismatch; a 60 min template suits a 50 min request better than 90 min
Equipment Feasibility 15% 100 if all slot patterns fillable by environment else 0 Hard gate; if the environment can't support the template's movement patterns, score is 0 (excluded)

Aggregate score formula:

final_score = ( goal_match * 0.25 + experience_match * 0.25 + days_fit * 0.20 + duration_fit * 0.15 + equipment_feasibility * 0.15 ) / 100

Example: User wants 4 days/week, 60 min sessions, strength focus, beginner level, home gym (dumbbells + barbell).

  • Upper/Lower (4 days, beginner, strength): Goal ✓ (100), Exp ✓ (100), Days perfect (100), Duration perfect (100), Equipment ✓ (100) → Score: 100%
  • Push/Pull/Legs (6 days, intermediate, muscle_gain): Goal ✗ (0), Exp ✗ (0), Days -2 distance (75), Duration perfect (100), Equipment ✓ (100) → Score: ~34%

Selection Scoring: Exercise Ranking Within a Slot

After filtering candidates for a slot, the engine ranks them by a multi-factor score. Unlike template matching, this is a normalized rank (candidates are compared pairwise, not as absolute 0–100 scales):

Rank Factor Priority Logic
Primary Muscle Match Tier 1 Exact match on slot's primary muscle > secondary > tertiary. Determines initial ranking
Difficulty Proximity Tier 1 Absolute difference from user's experience level; closer is better
Variety (Rotation Tracking) Tier 2 Penalize exercises used in recent weeks; reward variety
Slot Priority Tier 2 Primary slots prefer compound moves; secondary prefer isolations
Compound/Isolation Balance Tier 2 Track session compound count; if low, boost compound scores; if high, boost isolation
Unilateral/Bilateral Alternation Tier 3 Track recent unilateral/bilateral distribution; alternate to balance load

Variety & Rotation Formula

The engine maintains a rotation state to ensure variety across program weeks. For each exercise slot, it tracks:

  • Last used: Week number when this exercise was last picked
  • Rotation window: Typically 2–3 weeks; avoid repeating an exercise within this window

Variety score penalty:

weeks_since_last_use = current_week - last_used_week rotation_window = 3 if weeks_since_last_use < rotation_window: variety_penalty = (rotation_window - weeks_since_last_use) / rotation_window # Penalizes reuse; e.g., if last used 1 week ago and window is 3: # penalty = (3 - 1) / 3 = 0.67 (significant) else: variety_penalty = 0 # Exercise is old enough; no penalty

Candidates are ranked higher if they have been unused recently (penalty = 0). This ensures rotation while avoiding hard locks—if all alternatives are also within the window, the least-recent one wins.

Badge indicators for users: Exercises are labeled as New (never used), Rotated (reappearing after rotation window), or Repeated (within rotation window, typically from locking or fallback) in the UI preview.

6. Exercise Selection

For each slot, the engine selects an exercise by filtering and ranking:

Filter Stage

  • Candidate pool: Active exercises matching the slot's movement pattern or body region and muscle emphasis
  • Equipment: equipment_tags ⊆ environment.equipment_tags
  • Difficulty: difficulty_level ≤ user experience (with +1 level tolerance)
  • Contraindications: contraindications ∩ user injuries = ∅

Ranking Stage

After filtering, candidates are ranked by:

  • Primary-muscle fit
  • Difficulty proximity to user level
  • Variety (avoid repeating a movement already used in the program)
  • Slot priority (primary vs. secondary)
  • Compound/isolation balance (prefer compounds for primary slots, isolations for secondary)
  • Unilateral/bilateral alternation (balance single-leg/single-arm work across the session)

Constraints & Fallback

If a slot is locked or swapped, the forced exercise is selected. If a slot is excluded, that exercise is dropped. Otherwise, the top-ranked candidate wins. If the pool is empty after filters, the engine relaxes the least-critical filter (difficulty tolerance first) and flags the slot in the response.

7. Feedback & Re-adaptation

Re-adaptation is deterministic re-generation over the constraints overlay, never ad-hoc patching. Each user action mutates constraints, then selection re-runs only for affected slots, preserving everything locked.

Action Effect
Swap Replace one slot's exercise with a chosen valid alternative. Records is_user_swapped.
Exclude / regenerate slot Drop an exercise and let the engine pick the next best for that slot.
Adjust volume/intensity Nudge sets/day or program volume within safe bounds; re-applies via volume_adjustments.
Lock Pin a slot so re-adaptation never changes it.

Actions are repeatable until the user accepts the program.

8. API Endpoints

Endpoint Method Purpose
/programs/match POST Score templates, return ranked top 3 + required inputs
/programs/draft POST Create a draft program from a chosen template
/programs/{id}/preview GET Derive and return all weeks of a program
/programs/{id}/feedback POST Apply user feedback (swap/exclude/adjust/lock) and return updated draft
/programs/{id}/slots/{we_id}/alternatives GET List valid substitutes for a slot
/programs/{id}/accept POST Finalize program: set status=active, optionally archive prior active program
/programs/{id} GET Read-only program preview

9. Services Layout

All engine logic lives in backend/app/services/program/:

app/services/program/ matching.py # score templates vs profile -> ranked shortlist selection.py # pick exercises per slot (filters, ranking, constraints) progression/ __init__.py # registry keyed by model_key base.py # ProgressionModel protocol, SetScheme linear_load.py double_progression.py weekly_undulating.py deload.py # modifier/decorator drafting.py # template + selection -> persisted base-week draft adaptation.py # apply FeedbackAction -> mutate constraints -> re-run affected slots preview.py # derive concrete per-week schedule for display feedback_parser.py # FeedbackParser protocol (LLM-ready seam)

LLM-ready seam: The FeedbackParser protocol allows MVP to handle structured actions from the UI, while a future LLM implementation can parse natural-language feedback into the same FeedbackAction type without touching the core.

10. Testing & Performance

Performance

Generation runs on in-memory computation over ~135 exercises and a handful of templates. One cacheable exercise-library read. Generation is well under 100 ms versus the <2 s target. Async SQLAlchemy throughout; exercise library is cacheable in memory.

Testing (TDD, >80% coverage)

  • Unit tests: Match scoring, selection filters (equipment / injury / difficulty), each progression model's math, adaptation preserves locks
  • Integration tests: Full match → draft → feedback → accept flow
  • Determinism: Golden-file program snapshots validate reproducibility

11. Template Authoring & Roadmap

MVP (now): A small curated catalog of templates seeded via script, mirroring the exercise seed pattern. No authoring UI yet.

Initial examples (seeded):

  • Full Body (3 days, beginner, general/strength): Linear load + deload
  • Bodyweight Full Body (3 days, beginner, general): Double progression
  • Upper/Lower (4 days, intermediate, strength/muscle_gain): Double progression + deload
  • Push/Pull/Legs (6 days, intermediate–advanced, muscle_gain): Double progression + deload

Future (deferred, design forward-compatible):

  • Form-based template builder: A guided UI where trainers compose templates with dropdowns (never edit raw JSON/YAML)
  • LLM-assisted authoring: Describe a program in natural language; LLM emits a draft template that trainers refine
  • Ownership & roles: The MVP catalog is global/admin-curated. Trainer-owned or org-scoped templates with role/permission model are a future concern
Note: This engine is designed as a foundation. The three-layer template model (split + slot rules + progression reference) is forward-compatible with future features like form-based authoring and LLM-assisted design without reshaping the core architecture.
