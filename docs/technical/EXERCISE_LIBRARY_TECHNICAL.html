Overview

The program-generation engine (PROJECT_SCOPE.md's next major piece of work) needs something to select exercises from. The Exercise Library is that foundation: a read-only catalog of exercises tagged with everything the engine will eventually filter and substitute on — equipment required, muscle groups targeted, movement classification, difficulty, and injury contraindications — plus enough equipment variants of the same movement (barbell / dumbbell / band / machine / bodyweight versions of a bench press, for example) that the engine can swap one exercise for an equivalent when equipment or a limitation rules an option out.

This pass ships the data model, a schema-only migration, an idempotent content-seeding script, and a read-only API (list + get-by-id). It deliberately stops short of the substitution/alternatives logic itself — that's engine work, tracked separately.

Scope decision: UserProfile.injuries_limitations stays free text for now. Only Exercise.contraindications is tag-based in this pass; wiring free-text limitations to structured tags is deferred to the engine work.

Classification Methodology

Exercises are classified the way a trainer or physical therapist actually reasons about them — by movement pattern first, and by body region → muscle group second — rather than by the program-day split (push / pull / legs / core / cardio / flexibility) that FocusArea already uses for weekly scheduling (see Training Environments Technical Reference). Those are two different, deliberately independent taxonomies:

TaxonomyUsed forExample values
FocusAreaWhich day of a program a workout belongs topush, pull, legs, core, cardio, flexibility, full_body
MovementPatternWhich exercises are functionally interchangeable with each othersquat, hinge, horizontal_push, anti_rotation, carry, mobility…
Why movement pattern, not muscle group, is the primary substitution key

Two exercises that hit the "same muscle" can load a joint completely differently — a barbell back squat and a Romanian deadlift both heavily recruit the glutes and hamstrings, but a squat is knee-flexion-dominant while a hinge is hip-flexion-dominant. For someone flagged with a knee contraindication, "same muscle group" is the wrong substitution rule; "same movement pattern" is the right one, because it's what actually determines whether a joint is loaded. This is standard functional-training classification, most closely associated with the NASM Optimum Performance Training (OPT) model and the NSCA's movement-pattern-based exercise categorization — both organize resistance training around fundamental human movements (squat, hinge, lunge, push, pull, rotation, gait/locomotion) rather than isolated muscle targets. See References.

The Two Substitution Radii

Exercise carries two grouping fields at different granularities, so the future engine can try the narrowest safe swap first and widen only if needed:

  • movement_slug (narrow) — groups equipment variants of the same specific exercise. All four bench press variants (barbell, dumbbell, band, machine) share movement_slug="bench_press". Near-identical joint loading and skill demand; the safest possible swap.
  • movement_pattern (broad) — groups all exercises that train the same fundamental movement, regardless of specific exercise. Bench press variants and push-ups both share movement_pattern=HORIZONTAL_PUSH, even though push-ups don't share a movement_slug with any bench press variant. Functionally equivalent, but not the same skill/loading profile.

Example: a user with only dumbbells and a knee contraindication needing a horizontal push exercise → the engine tries movement_slug="bench_press" first (finds Dumbbell Bench Press, no swap in exercise character needed), and would only widen to the full movement_pattern=HORIZONTAL_PUSH pool if no bench_press variant matched their equipment.

Body Region → Muscle Group

Muscle-level detail exists as a secondary axis for balance/volume reasoning (e.g. "make sure the program hits posterior chain, not just quads"), organized top-down: BodyRegion (upper_body / lower_body / core / full_body) contains ALLOWED_MUSCLE_GROUPS entries. Deltoids are deliberately split into shoulders_anterior / shoulders_lateral / shoulders_posterior rather than one generic shoulders tag — this is exactly the granularity a trainer uses to balance push/pull volume, and it's what lets a shoulder contraindication or an anterior-delt-overuse flag mean something precise (an overhead press loads anterior/lateral delts; a face pull targets posterior delts specifically).

Architecture

Exercise (table: exercises) ├── slug (unique) — e.g. "barbell-bench-press" ├── movement_slug (indexed) — e.g. "bench_press" [narrow substitution radius] ├── body_region (enum) — upper_body | lower_body | core | full_body ├── movement_pattern (enum) — squat | hinge | lunge | horizontal_push | ... [broad substitution radius] ├── primary_muscles / secondary_muscles (JSON list[str], validated against ALLOWED_MUSCLE_GROUPS) ├── equipment_tags (JSON list[str], validated against ALLOWED_EQUIPMENT_TAGS — shared with TrainingEnvironment) ├── difficulty_level (enum — reuses User's ExperienceLevel, not redefined) ├── contraindications (JSON list[str], validated against ALLOWED_CONTRAINDICATION_TAGS) ├── instructions, form_cues[], safety_notes └── is_active (bool) — deactivate, never hard-delete; future Workout/UserWorkoutLog rows may reference an id No FK to users — exercises are a shared, global library, not user-owned data.

Constants

Equipment tags are shared with TrainingEnvironment on purpose. Reusing ALLOWED_EQUIPMENT_TAGS as-is (rather than a separate exercise-equipment vocabulary) means an exercise's equipment_tags can be checked directly against a TrainingEnvironment.equipment_tags set with no translation layer. The ez_bar / machine / medicine_ball / ab_wheel / sled tags were added to this shared list (and mirrored into frontend/src/types/trainingEnvironment.ts's equipment picker) specifically to support machine- and implement-based exercise variants added when the library was expanded from 48 to 135 exercises — an exercise tagged for equipment nobody can select in the training-environment picker would be permanently unreachable by the equipment filter.

Migration

Revision a1b2c3d4e5f6 (down_revision = "77fcb49685a6") is schema only — no data. Two new Postgres enum types are created (bodyregion, movementpattern); difficulty_level reuses the existing experiencelevel enum type (created by ebd4d0a4d103_initial_schema.py for UserProfile.experience_level) rather than recreating it.

⚠️ Reusing an existing Postgres enum inside op.create_table silently tries to recreate it

The obvious approach — sa.Enum("BEGINNER", "INTERMEDIATE", "ADVANCED", name="experiencelevel", create_type=False) — still emits CREATE TYPE experiencelevel ... and fails with DuplicateObject. Root cause: Alembic's op.create_table calls table.dispatch.before_create(table, connection, checkfirst=False, ...), and the generic sa.Enum's creation-guard logic only skips creation when the type has bound metadata or checkfirst is True — neither is true for a fresh, unbound table inside a single op.create_table call, so create_type=False is silently ignored on the generic type. The fix is to use the dialect-specific sqlalchemy.dialects.postgresql.ENUM class directly for any column referencing an already-existing type — its create_type=False is honored in this exact scenario. New types (bodyregion, movementpattern) still use plain sa.Enum since they need to be created.

from sqlalchemy.dialects import postgresql body_region_enum = sa.Enum("UPPER_BODY", "LOWER_BODY", "CORE", "FULL_BODY", name="bodyregion") movement_pattern_enum = sa.Enum("SQUAT", "HINGE", ... , name="movementpattern") # Must be the postgresql-dialect ENUM — only it honors create_type=False here. experience_level_enum = postgresql.ENUM( "BEGINNER", "INTERMEDIATE", "ADVANCED", name="experiencelevel", create_type=False ) def upgrade() -> None: op.create_table( "exercises", ..., sa.Column("body_region", body_region_enum, nullable=False), sa.Column("movement_pattern", movement_pattern_enum, nullable=False), sa.Column("difficulty_level", experience_level_enum, nullable=False), ..., ) op.create_index(op.f("ix_exercises_slug"), "exercises", ["slug"], unique=True) op.create_index(op.f("ix_exercises_movement_slug"), "exercises", ["movement_slug"], unique=False) def downgrade() -> None: op.drop_index(...); op.drop_table("exercises") # experiencelevel is NOT dropped — user_profiles still depends on it. body_region_enum.drop(op.get_bind(), checkfirst=True) movement_pattern_enum.drop(op.get_bind(), checkfirst=True)

Verified with a live round-trip: alembic upgrade head → alembic downgrade -1 → alembic upgrade head, confirming the table, both new enum types, and the two indexes all clean up and recreate correctly, and that experiencelevel is untouched throughout.

CRUD & Filtering Semantics

File: backend/app/crud/exercise.py. list_exercises always filters is_active.is_(True), applies body_region / movement_pattern / difficulty_level as SQL .where(...) equality (cheap, indexed/enum columns), then fetches once and applies muscle_group / equipment_tags / exclude_contraindications as Python-side filtering on the JSON list columns — this keeps the same query portable across the SQLite test database and Postgres, since JSON containment operators differ between the two dialects.

result = await db.execute(query) exercises = list(result.scalars().all()) if muscle_group is not None: exercises = [ex for ex in exercises if muscle_group in ex.primary_muscles or muscle_group in ex.secondary_muscles] if equipment_tags: equipment_set = set(equipment_tags) # exercise's full equipment requirement must be satisfiable by what the caller has — # NOT "caller's tags must be a subset of the exercise's tags" (that's backwards). exercises = [ex for ex in exercises if set(ex.equipment_tags).issubset(equipment_set)] if exclude_contraindications: contraindications_set = set(exclude_contraindications) exercises = [ex for ex in exercises if not contraindications_set.intersection(set(ex.contraindications))]
⚠️ equipment_tags subset check direction is easy to get backwards

The filter answers "what can I do with the equipment I actually have?" — so it must check that the exercise's equipment requirement is a subset of the caller's tags, not the other way around. The inverted form (requested_tags.issubset(exercise.equipment_tags)) reads plausibly but means "the exercise must require at least these tags," which returns exercises the caller cannot actually perform — e.g. querying equipment_tags=dumbbells would incorrectly return a Bulgarian split squat that also requires a bench. Caught during review by querying the live seed data and manually checking whether every returned exercise's tags were truly satisfiable by the single requested tag.

Seed Content Pipeline

Migrations are meant to be immutable once applied — baking every future exercise add/remove/edit into a new schema migration would put routine content curation on the same ceremony as schema evolution. So content is split from schema entirely:

1. Migration (a1b2c3d4e5f6) → creates the empty "exercises" table + indexes. Runs once, immutable. 2. exercises.py (seed data) → EXERCISE_SEED_DATA: list[dict], 135 exercises. Plain Python, edited anytime. 3. seed_exercises.py → upsert_exercises(db): idempotent insert/update/deactivate by slug. 4. seed-exercises.yml → workflow_dispatch-only GitHub Action to re-run step 3 against the live ECS service.

File: backend/app/db/seed/seed_exercises.py

async def upsert_exercises(db: AsyncSession) -> None: seed_slugs: set[str] = {str(ex["slug"]) for ex in EXERCISE_SEED_DATA} for data in EXERCISE_SEED_DATA: existing = await db.execute(select(Exercise).where(Exercise.slug == data["slug"])) exercise = existing.scalar_one_or_none() if exercise is None: db.add(Exercise(**data, is_active=True)) # new slug → insert else: for key, value in data.items(): setattr(exercise, key, value) exercise.is_active = True # existing slug → update in place await db.flush() # any row whose slug is no longer in the seed list gets deactivated, never deleted — # future Workout / UserWorkoutLog rows may reference it by id. existing_exercises = await db.execute(select(Exercise)) for exercise in existing_exercises.scalars().all(): if exercise.slug not in seed_slugs and exercise.is_active: exercise.is_active = False await db.commit()

Runnable standalone: python -m app.db.seed.seed_exercises (backed by an async def main() using the app's own async_session).

Content Coverage

backend/app/db/seed/exercises.py currently holds 135 exercises across all 13 MovementPattern values, each specific movement represented by its equipment variants (e.g. the four bench-press variants, ten squat-pattern variants from bodyweight through Smith-machine and box squat). Every entry is validated in tests against the three ALLOWED_* constants, and no two entries share a slug.

Modeling rule enforced during the 48 → 135 expansion: a single row's equipment_tags lists equipment that must all be present simultaneously (e.g. Bulgarian Split Squat genuinely needs ["dumbbells", "bench"] together). Equipment that represents alternative ways to do a movement (e.g. Romanian Deadlift with a barbell or with dumbbells) must be split into separate rows sharing one movement_slug — combining them into one row's tag list would incorrectly require the caller to own both, given the AND semantics of the equipment_tags subset check described above.

Testing

File: backend/tests/test_exercises.py (16 tests), reusing the async client / DB fixtures from backend/tests/conftest.py.

  • Seed data integrity — every equipment_tags / primary_muscles / secondary_muscles / contraindications value in the seed list is checked against its ALLOWED_* constant, catching typos before they ever reach the database.
  • upsert_exercises idempotency — running it twice produces the same row count with no duplicate slugs; changing a field and re-running updates the row in place; removing a slug and re-running sets is_active=False without deleting the row.
  • API filtering — GET /exercises respects every filter param individually and combined, and excludes is_active=False rows.
  • 404 / 401 — missing id via ExerciseNotFoundError, unauthenticated requests on both routes.

Result at last verification: backend 61/61 pytest passing (ruff, black, mypy clean); frontend 66/66 vitest passing (eslint, tsc clean) after the equipment-tag picker sync; migration round-trip verified against live Postgres; seed script re-run confirmed idempotent (135 rows, unchanged on re-run).

Deployment Considerations

Applying the Migration

docker-compose exec backend alembic upgrade head

Seeding / Updating Content

Locally, run once after migrating:

docker-compose exec backend python -m app.db.seed.seed_exercises

In production, content changes don't require a full deploy. .github/workflows/seed-exercises.yml is a separate, workflow_dispatch-only Action (no push trigger) that reuses deploy.yml's aws ecs run-task pattern against the already-running ECS service, overriding only the container command:

--overrides '{"containerOverrides":[{"name":"backend","command":["uv","run","python","-m","app.db.seed.seed_exercises"]}]}'

Anyone with repo write access can trigger it manually from the GitHub Actions UI after merging a change to exercises.py, independent of code deploys. See Infrastructure & Deployment Technical Reference for how the underlying ECS task-run mechanism works.

Troubleshooting

❌ GET /exercises?equipment_tags=X returns an exercise I don't have the equipment for

See the CRUD & Filtering section above — confirm list_exercises checks set(ex.equipment_tags).issubset(equipment_set), not the reverse.

❌ Alembic migration fails with DuplicateObject: type "experiencelevel" already exists

A column is using generic sa.Enum(..., create_type=False) for a type that already exists. Switch to sqlalchemy.dialects.postgresql.ENUM(..., create_type=False) — see the Migration section above.

❌ A seed content change didn't show up after editing exercises.py

Editing the file alone doesn't touch the database — re-run python -m app.db.seed.seed_exercises locally, or trigger the Seed Exercises GitHub Action for production.
