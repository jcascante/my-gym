1. Overview

This reference audits every field across MyGym's backend models and states, as of the current codebase, whether it is actually read by the program-matching or program-generation engine — not just whether it is collected and stored. It exists because onboarding, profile editing, and program creation collect data from three separate places (UserProfile, TrainingEnvironment, and per-request payloads), and not everything collected is consumed downstream.

This is a current-state audit, not a proposal. Fields marked unused or partially used are documented as-is; no schema or behavior changes are implied by this page.

Legend

✅ UsedRead by the matching engine (services/program/matching.py) and/or the selection/drafting engine (services/program/selection.py, drafting.py)
⚠️ PartialRead by some part of the pipeline but not all of it, or duplicated/shadowed by a separate per-request field

3. UserProfile

Collected across the onboarding wizard's 6 steps (frontend/src/pages/OnboardingPage.tsx). This is the table with the largest gap between what's collected and what's consumed.

FieldStatusUsed byPurpose / how it's used
age ❌ Unused — Collected in onboarding step 1. No service reads it — not in matching.py, selection.py, or drafting.py.
gender ❌ Unused — Same as above — free-text/select value, never read downstream.
weight_kg ❌ Unused — Collected but not used for load estimation, BMI, or anything else. No consumer.
height_cm ❌ Unused — Same — no consumer.
activity_level ❌ Unused — Collected in onboarding step 2, alongside experience_level. Unlike experience_level, it is never read by any service.
fitness_focus ✅ Used Matching (goal factor) ProgramBuilderPage.tsx reads userProfile.fitness_focus (fallback "general") to build MatchRequest.fitness_focus, which feeds the goal factor in rank_templates() — currently 25% of the match score.
experience_level ✅ Used Matching (experience factor) + Selection (difficulty filter/ranking) Read directly from user.profile.experience_level in the /programs/match endpoint and in _ctx_for(). Feeds template scoring and gates/ranks exercise candidates by difficulty_level in selection.py.
days_per_week ⚠️ Partial Matching (days factor) — but only via a re-entered value Stored on the profile, but the program-creation wizard (ProgramCreationForm.tsx) does not read it — it always defaults to 3 and asks the user again. The value that actually reaches MatchInput.days_per_week is whatever the user re-enters in the wizard, not the profile field.
workout_duration_min ⚠️ Partial Matching (duration factor) — same caveat Same issue as days_per_week: the wizard defaults session_duration_min to 60 and ignores the stored profile value.
injuries_limitations ⚠️ Partial Selection/drafting only (contraindication filter) Free text, split on commas and lowercased in _ctx_for() (api/v1/endpoints/programs.py), then matched literally against each exercise's contraindications tags (knee, shoulder, lower_back, wrist, elbow, hip, neck, ankle). Only built for /draft, /feedback, and /alternatives — the /match endpoint never builds a SelectionContext, so injuries have no effect on template ranking, only on which exercise is picked once a template is already chosen. Matching is also fragile: phrasing like "bad knee" doesn't split into a token that matches "knee".
short_term_goals ❌ Unused — Free text collected in onboarding step 5. Not read by any backend service, and not displayed anywhere in the frontend after submission.
medium_term_goals ❌ Unused — Same as short_term_goals — write-only.

4. TrainingEnvironment

Collected via the "Training Environments" onboarding step and the standalone environment CRUD (TrainingEnvironmentForm). One user may have several; each program is generated against one chosen environment.

FieldStatusUsed byPurpose / how it's used
name ✅ UsedFrontend display Label shown in environment pickers; not read by matching/selection.
environment_type ❌ Unused — Categorization (commercial_gym, home, bodyweight, …) shown in the UI, but no backend service branches on it — only equipment_tags drives feasibility/filtering.
equipment_tags ✅ Used Matching (template feasibility gate) + Selection (equipment filter) Passed as MatchInput.environment_equipment and SelectionContext.equipment. A template is only feasible if every required slot can be filled by an exercise whose equipment_tags are a subset of this list (template_is_feasible() in selection.py).
is_default ⚠️ Partial Frontend only Used client-side to pre-select an environment in the UI; not read by any backend generation logic.

5. Program-Creation-Only Inputs (not on UserProfile)

These fields exist only in MatchRequest/DraftRequest (schemas/program_api.py) — re-entered every time a program is created, then persisted into WorkoutProgram.constraints once accepted. They have no profile-level counterpart, so there's no duplication concern for them.

FieldStatusUsed byPurpose
movement_preferences✅ UsedMatching (movement_preference factor, 15%) + Selection (exercise ranking)Per-equipment-family weight slider (0–2.0); biases which candidate exercise wins a slot.
complementary_focus✅ UsedMatching (focus_complement factor, 12%) + SelectionWhether accessory/isolation slots favor muscle-coverage diversity.
progression_style✅ UsedMatching (periodization factor, 8%) + draftingconsistent vs variable weekly load progression; scores templates whose progression model matches the requested style.
effort_method✅ UsedDrafting/previewHow intensity is displayed/computed (rpe, rir, borg, percent_1rm); percent_1rm multiplies a manually-entered required input by the scheme's intensity_pct.
variety_preference✅ UsedDrafting (rotation pool size)Controls how many alternates are kept per slot for weekly rotation.
weight_unit, duration_weeks✅ UsedDraftingDisplay unit and program length; not scoring inputs.
required_inputs✅ UsedDrafting (_base_load_for)Per-template, per-movement-pattern starting load, manually entered by the user via RequiredInputsForm — the only place a numeric starting weight enters the system today (weight_kg is not used for this).

7. WorkoutProgram / Workout / WorkoutExercise / Exercise

These are the engine's output, not input — included here only to close the loop. WorkoutProgram.constraints (JSON) is where every per-request, creation-time-only field from Section 5 ends up persisted once a program exists, since UserProfile never stores them. Exercise is catalog data (equipment tags, contraindications, muscles, difficulty) that every profile/environment field above is ultimately filtered or scored against — see Exercise Library Technical Reference for its full field list.

8. Summary of Gaps

#GapFields involved
1Collected twice, profile value ignored — wizard hardcodes its own defaults instead of reading the stored profile valuedays_per_week, workout_duration_min
2Free-text field parsed with fragile comma-split/lowercase matching against a fixed vocabularyinjuries_limitations
3Only enforced after a template is already chosen, never scored during matching itselfinjuries_limitations
4Captured biometrics with zero downstream consumersage, gender, weight_kg, height_cm, activity_level
5Write-only free text — stored, never read or displayedshort_term_goals, medium_term_goals
Documentation drift note: Section 5 of the Program Generation Engine doc lists match weights as Goal 25% / Experience 25% / Days 20% / Duration 15% / Equipment 15%. The current code (MatchWeights in services/program/matching.py) is Goal 25% / Experience 20% / Days 12% / Duration 8% / Movement preference 15% / Focus complement 12% / Periodization 8% — three factors (movement_preference, focus_complement, periodization) exist in code but aren't reflected in that doc's weight table. This page uses the current code's weights.
Scope: This page documents current behavior only. A redesign addressing the gaps above (wiring biometrics into load/recovery estimation, structuring injuries, prefilling the creation wizard from the profile, scoring injuries during matching) is being scoped separately and will land as its own spec.
