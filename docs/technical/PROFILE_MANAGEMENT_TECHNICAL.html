Overview

The Profile Management system handles user profile data (fitness level, goals, preferences) with support for partial updates and explicit null-value persistence. Users can update their profile incrementally — changing only specific fields while preserving all others — or explicitly clear fields by setting them to null.

Key Features

  • Partial Updates: Only explicitly provided fields are updated; unchanged fields are preserved
  • Null-Value Persistence: Users can explicitly set profile fields to null/empty to clear them
  • Type Safety: Pydantic schemas ensure data validation and type consistency
  • Async Operations: All database operations use async SQLAlchemy for non-blocking I/O
  • Comprehensive Testing: Full test coverage for edge cases (partial updates, null persistence, field preservation)

Architecture

The profile management system follows a layered architecture with clear separation of concerns:

Frontend (React) ↓ ProfilePage.tsx (Form validation & normalization) ↓ saveUserProfile() API client ↓ Backend HTTP Layer ↓ POST /api/v1/users/profile ├→ Request validation (Pydantic UserProfileRequest) ├→ Partial data extraction (exclude_unset=True) ↓ CRUD Layer (app/crud/user.py) ├→ create_or_update_user_profile() ├→ Existing profile lookup ├→ Field-by-field setattr (including null values) ↓ Database Layer └→ UserProfile model (SQLAlchemy ORM)

Key Design Decisions

  • Pydantic exclude_unset=True: Only serializes fields that were explicitly set in the request, distinguishing "not provided" from "explicitly null"
  • CRUD loop with unconditional setattr: Sets all provided fields including null values, allowing explicit clears without special logic
  • Frontend form normalization: Parse helpers convert empty strings to null before sending, ensuring consistent representation
  • Optional nullable columns: Database schema allows NULL for all profile fields, enabling flexible data deletion

API Endpoints

POST /api/v1/users/profile

Purpose: Create or update the current user's profile with partial data

Authentication: Required (HTTP-only JWT cookie)

Request Body

{ "age": 30 | null, "gender": "male" | "female" | "other" | null, # Any string accepted "weight_kg": 75.5 | null, "height_cm": 175.5 | null, # Float: decimals allowed "activity_level": "sedentary" | "lightly_active" | ... | null, "fitness_focus": "strength" | "endurance" | ... | null, "experience_level": "beginner" | "intermediate" | "advanced" | null, "days_per_week": 3 | null, "workout_duration_min": 60 | null, "equipment": "home" | "gym" | "bodyweight" | null, "injuries_limitations": "Back pain, knee injury" | null, "short_term_goals": "Build consistency" | null, "medium_term_goals": "Gain muscle" | null }

Partial Update Example

To update only age and weight, send:

{ "age": 31, "weight_kg": 78.5 }

Result: Only age and weight_kg are updated. All other fields (gender, height, etc.) are preserved as-is in the database.

Null-Value Clear Example

To clear the fitness_focus field:

{ "fitness_focus": null }

Result: fitness_focus is set to NULL in the database. All other fields are preserved.

Response

{ "id": 1, "email": "user@example.com", "first_name": "John", "last_name": "Doe", "profile": { "id": 1, "age": 31, "gender": "male", "weight_kg": 78.5, // ... all other fields } }

Status Codes

Code Meaning
200 Profile created or updated successfully
401 Unauthorized (no valid JWT token)
422 Validation error (invalid enum value, type mismatch, etc.)

Endpoint Handler

@router.post("/profile", response_model=UserWithProfileResponse) async def create_update_profile_endpoint( profile_data: UserProfileRequest, current_user: User = Depends(get_current_user), db: AsyncSession = Depends(get_db), ) -> User: # exclude_unset=True: only fields explicitly provided by client # Omitted fields remain unset → CRUD won't overwrite them await create_or_update_user_profile( db, current_user.id, profile_data.model_dump(exclude_unset=True) ) # Refresh relationship to get updated profile await db.refresh(current_user, attribute_names=["profile"]) return current_user

CRUD Operation

File: backend/app/crud/user.py

async def create_or_update_user_profile( db: AsyncSession, user_id: int, profile_data: dict[str, Any], ) -> UserProfile: existing_profile = await get_user_profile(db, user_id) if existing_profile: # Update existing: set each field in profile_data # This includes null values (allows explicit clears) for key, value in profile_data.items(): setattr(existing_profile, key, value) db.add(existing_profile) else: # Create new profile with provided data existing_profile = UserProfile(user_id=user_id, **profile_data) db.add(existing_profile) await db.flush() await db.commit() await db.refresh(existing_profile) return existing_profile

Frontend Validation

File: frontend/src/pages/ProfilePage.tsx

# Parse helpers: convert form values to proper types or null const parseNumOrNull = (val: string | number | null | undefined): number | null => { if (val === '' || val === null || val === undefined) return null; const num = parseInt(val.toString()); return Number.isNaN(num) ? null : num; }; const parseFloatOrNull = (val: string | number | null | undefined): number | null => { if (val === '' || val === null || val === undefined) return null; const num = parseFloat(val.toString()); return Number.isNaN(num) ? null : num; }; const parseStringOrNull = (val: string | null | undefined): string | null => { const trimmed = val?.toString().trim(); return trimmed && trimmed !== '' ? trimmed : null; }; # Apply to all form fields before sending const profileData: UserProfile = { age: parseNumOrNull(formData.age), # Integer: no decimals gender: parseStringOrNull(formData.gender), weight_kg: parseFloatOrNull(formData.weight_kg), # Float: decimals allowed height_cm: parseFloatOrNull(formData.height_cm), # Float: decimals allowed # ... rest of fields };

Data Flow

Scenario 1: Partial Update (Update age only)

1. User clicks "Edit Profile" → Form loads with current profile data 2. User changes age from 30 to 31 → formData.age = "31" (string from input) → Other fields unchanged 3. User clicks "Save Changes" → handleSubmit fires → parseNumOrNull("31") → 31 (number) → profileData = {age: 31} (only changed field) 4. saveUserProfile(profileData) POST /api/v1/users/profile Request body: { "age": 31 } 5. Backend receives request → Pydantic validates: ✓ age is valid int → profile_data.model_dump(exclude_unset=True) → Returns {age: 31} (only age was set in request) 6. CRUD: create_or_update_user_profile(db, user_id, {age: 31}) → Lookup existing profile → for key, value in {age: 31}.items(): → setattr(profile, key, value) → setattr(profile, "age", 31) ← Only this executes → gender, weight, height, etc. UNTOUCHED 7. Database UPDATE UPDATE user_profiles SET age = 31, updated_at = now() WHERE user_id = 1 Result: age=31, gender="male" (preserved), weight_kg=75.5 (preserved) 8. Response returns full profile with all fields

Scenario 2: Null-Value Clear (Clear fitness_focus)

1. User wants to clear their fitness_focus → Selects empty option in fitness_focus dropdown → formData.fitness_focus = "" (empty string) 2. User clicks "Save Changes" → handleSubmit fires → parseStringOrNull("") → null (empty strings become null) → profileData = {fitness_focus: null} (only changed field) 3. saveUserProfile(profileData) POST /api/v1/users/profile Request body: { "fitness_focus": null } 4. Backend receives request → Pydantic validates: ✓ fitness_focus: Optional[FitnessFocus] → profile_data.model_dump(exclude_unset=True) → Returns {fitness_focus: null} (only fitness_focus was set) 5. CRUD: create_or_update_user_profile(db, user_id, {fitness_focus: null}) → Lookup existing profile → for key, value in {fitness_focus: null}.items(): → setattr(profile, key, value) → setattr(profile, "fitness_focus", None) ← Explicitly set to null! → Other fields UNTOUCHED 6. Database UPDATE UPDATE user_profiles SET fitness_focus = NULL, updated_at = now() WHERE user_id = 1 Result: fitness_focus=NULL (cleared), weight_kg=75.5 (preserved) 7. Response returns profile with fitness_focus=null

Null Handling Strategy

Key Principle: Partial Updates The backend uses exclude_unset=True to distinguish:
  • Not provided (omitted from request) → Field is preserved
  • Explicitly null (sent as null) → Field is cleared

Frontend Normalization

All empty form inputs are converted to null before sending to the backend:

  • "" (empty string) → null
  • Missing/undefined → null
  • Valid values → parsed and sent as-is (e.g., "30" → 30)

Database Constraints

All UserProfile columns are nullable in SQLAlchemy 2.0+:

age: Mapped[int | None] = mapped_column(Integer) weight_kg: Mapped[float | None] = mapped_column(Float) # ... all fields are nullable (Mapped[T | None])

This allows:

  • First-time profile creation with partial data
  • Users explicitly clearing fields without cascading deletes
  • Flexible profile evolution over time

Edge Cases Handled

❓ What if user sends empty dict {}?

No fields are set in Pydantic, so exclude_unset=True returns empty dict. CRUD receives empty dict, no fields are updated. All existing data preserved. ✓

❓ What if user sends all fields explicitly?

All fields have exclude_unset=False, so all are included even if unchanged. CRUD updates all fields. Safe because we're setting to the same values. ✓

❓ What if frontend sends invalid enum value?

Pydantic validation catches it before CRUD. Returns 422 Unprocessable Entity. Database never receives invalid data. ✓

Troubleshooting

❌ User saves profile, but unchanged fields get cleared

Cause: Frontend is sending all 13 fields with null for unchanged fields. Backend's setattr is setting them to NULL instead of preserving.

Fix: Ensure backend uses profile_data.model_dump(exclude_unset=True) to only include fields that were explicitly set in the request. See: backend/app/api/v1/endpoints/users.py:24

❌ User tries to clear a field (set to null), but it's not being cleared

Cause: Frontend is sending undefined/NaN instead of null for empty fields. Backend filter might be excluding None values.

Fix: Ensure parse helpers convert empty inputs to null: - parseNumOrNull("") → null - parseStringOrNull("") → null

❌ API returns 422 Unprocessable Entity

Cause: Invalid enum value sent (e.g., "super_strength" for fitness_focus).

Fix: Validate form inputs against allowed enum values. Check backend/app/schemas/user.py for valid values.

❌ Test: "gender expected 'male', got None"

Cause: Partial update test is not using exclude_unset=True. CRUD is receiving all fields including unset ones as None.

Fix: Verify CRUD layer at backend/app/crud/user.py:50-52 only has setattr(existing_profile, key, value) without if value is not None.

⚠️ Decimal heights truncated (175.5 → 175)

Cause: Frontend ProfilePage.tsx:81 uses parseNumOrNull (integer parsing) for height_cm instead of parseFloatOrNull. This truncates decimal heights.

Fix: Change line 81 from height_cm: parseNumOrNull(formData.height_cm) to height_cm: parseFloatOrNull(formData.height_cm).
