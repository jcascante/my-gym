⚙️ Training Environments & Program Creation Technical Documentation

Multi-environment data model, CRUD API, and the program-generation request placeholder

Overview Architecture API Endpoints Models & Schemas Migration Implementation Frontend Data Flow Testing Deployment Troubleshooting

Overview

Users don't just have one training setup. The same person might lift at a commercial gym on weekdays, train bodyweight-only at home on weekends, and occasionally drop into a CrossFit box. Rather than force a single coarse equipment value onto UserProfile, equipment and location are modeled as their own first-class, multi-valued resource: TrainingEnvironment.

This introduces a three-tier data model:

TierResourceChangesPersisted?
1UserProfileRarely (age, experience, goals)Yes — one per user
2TrainingEnvironmentOccasionally (new gym, new equipment)Yes — many per user
3ProgramCreationRequestEvery time a program is generatedNo — one-shot request payload

This work covers gathering and saving tiers 1 and 2, plus a placeholder endpoint (POST /api/v1/programs) that validates the full tier-3 payload and returns 501 Not Implemented — real program generation is intentionally out of scope.

Note: UserProfile.equipment was removed outright rather than kept alongside TrainingEnvironment — the codebase was pre-production (single initial migration, no seed data), so there was no need to support both concepts simultaneously.

Architecture

User ├── UserProfile (1:1 — stable personal/fitness data) └── TrainingEnvironment (1:N — where/with-what, reusable across program attempts) id, user_id, name, environment_type, equipment_tags[], is_default ProgramCreationRequest (not a table — validated request body only) references: environment_id → TrainingEnvironment fields: days_per_week, preferred_days[], session_duration_min, start_date, focus_areas[], weight_unit, available_weight_increments[], progression_style

API Endpoints

Training Environments CRUD

GET /api/v1/training-environments

List all environments owned by the current user.

POST /api/v1/training-environments

Create a new environment. Returns 201.

{ "name": "Home Gym", "environment_type": "home", "equipment_tags": ["dumbbells", "pull_up_bar"], "is_default": true }
GET /api/v1/training-environments/{environment_id}

Fetch one environment, scoped to the current user. 404 if missing or owned by someone else.

PATCH /api/v1/training-environments/{environment_id}

Partial update (only provided fields change), same exclude_unset pattern used by UserProfileRequest.

DELETE /api/v1/training-environments/{environment_id}

Returns 204 on success.

Program Creation Placeholder

POST /api/v1/programs Stub — returns 501

Validates ownership of environment_id and the full preferences payload, then unconditionally raises HTTPException(501). Demonstrates the intended request shape without generating anything.

{ "environment_id": 1, "days_per_week": 4, "preferred_days": ["monday", "wednesday", "friday", "saturday"], "session_duration_min": 60, "start_date": "2026-08-01", "focus_areas": ["push", "pull", "legs"], "weight_unit": "kg", "available_weight_increments": [1.25, 2.5, 5], "progression_style": "consistent" }
CodeMeaning
422Malformed payload (Pydantic validation, before the handler runs)
404environment_id missing or not owned by the current user
501Payload is valid and owned — program generation just isn't built yet
401Unauthenticated

Models & Schemas

TrainingEnvironment (SQLAlchemy)

File: backend/app/models/training_environment.py

class EnvironmentType(str, enum.Enum): COMMERCIAL_GYM = "commercial_gym" HOME = "home" BODYWEIGHT = "bodyweight" CROSSFIT_BOX = "crossfit_box" OTHER = "other" class TrainingEnvironment(Base): __tablename__ = "training_environments" id: Mapped[int] = mapped_column(Integer, primary_key=True) user_id: Mapped[int] = mapped_column(ForeignKey("users.id"), nullable=False, index=True) name: Mapped[str] = mapped_column(String(100), nullable=False) environment_type: Mapped[EnvironmentType] = mapped_column(Enum(EnvironmentType), nullable=False) equipment_tags: Mapped[list[str]] = mapped_column(JSON, nullable=False, default=list) is_default: Mapped[bool] = mapped_column(Boolean, nullable=False, default=False) created_at: Mapped[datetime] = mapped_column(DateTime, default=_utcnow, nullable=False) updated_at: Mapped[datetime] = mapped_column(DateTime, default=_utcnow, onupdate=_utcnow, nullable=False) user: Mapped["User"] = relationship(back_populates="environments")
Why JSON, not an enum column? equipment_tags is a flexible list[str] stored as JSON rather than a fixed SQL enum, validated against a Python allow-list (ALLOWED_EQUIPMENT_TAGS in backend/app/core/constants.py) via a Pydantic field_validator. This works identically on SQLite (the test database) and PostgreSQL (production) — a native Postgres array/enum column would require dialect-specific handling in tests.

Program Creation Schema (Pydantic only — no table)

File: backend/app/schemas/program.py

class FocusArea(str, enum.Enum): PUSH = "push"; PULL = "pull"; LEGS = "legs"; CORE = "core" CARDIO = "cardio"; FLEXIBILITY = "flexibility"; FULL_BODY = "full_body" class WeightUnit(str, enum.Enum): KG = "kg"; LBS = "lbs" class ProgressionStyle(str, enum.Enum): CONSISTENT = "consistent" # same sets/reps/weight jumps every week VARIABLE = "variable" # undulating, varies across days/weeks class ProgramCreationRequest(BaseModel): environment_id: int days_per_week: int preferred_days: list[DayOfWeek] session_duration_min: int start_date: date focus_areas: list[FocusArea] = [] weight_unit: WeightUnit available_weight_increments: list[float] = [] progression_style: ProgressionStyle

focus_areas intentionally reuses the same vocabulary PROJECT_SCOPE defines for the (unbuilt) Exercise library, so it stays shared once real exercises exist.

CRUD Layer — Default Exclusivity

File: backend/app/crud/training_environment.py

Exactly one environment per user can be is_default=True. This is enforced at the CRUD layer rather than a database constraint (a partial unique index behaves differently between SQLite and Postgres):

async def _clear_existing_default(db, user_id, exclude_id=None): # Unsets any other default for this user before a new one is set ...

Both create_training_environment and update_training_environment call this whenever is_default=True is being set.

Migration

Revision 77fcb49685a6 (down_revision = "ebd4d0a4d103") does two things: drops UserProfile.equipment and creates training_environments.

⚠️ Alembic autogenerate doesn't drop orphaned Postgres enum types

op.drop_column("user_profiles", "equipment") alone leaves the old equipment enum type in the database, which then blocks recreating a column with that name. Fixed by explicitly calling equipment_enum.drop(op.get_bind(), checkfirst=True) in upgrade(), and .create(...) in downgrade() (along with dropping the new environmenttype enum on downgrade). Verified with a live round-trip: alembic upgrade head → alembic downgrade -1 → alembic upgrade head.

docker-compose exec backend alembic upgrade head docker-compose exec backend alembic downgrade -1 # lossy for environment data — acceptable pre-production docker-compose exec backend alembic upgrade head

⚠️ Nested <form> is invalid HTML

TrainingEnvironmentForm is rendered inside OnboardingPage's own <form> for the onboarding step. An earlier version used a native <form onSubmit> internally, which produced a DOM warning ("form cannot be a descendant of form") and invalid markup. Fixed by making it a plain <div> with a type="button" onClick={handleSubmit} submit button instead of native form submission. A regression test asserts container.querySelector('form') is null for this component.

⚠️ err instanceof AxiosError is unreliable in component tests

ProgramCreationForm needs to distinguish the expected 501 from other errors. Checking err instanceof AxiosError works against a real backend but fails in Vitest component tests that mock rejections with plain objects. Fixed with a duck-typed check instead:

const status = (err as { response?: { status?: number } })?.response?.status; if (status === 501) { /* coming-soon message */ } else { /* generic error */ }

Data Flow

Adding an Environment (from either Onboarding or the Environments page)

1. User fills TrainingEnvironmentForm, clicks "Save Environment" 2. Frontend POST /api/v1/training-environments {name, environment_type, equipment_tags, is_default} 3. Backend validates equipment_tags against ALLOWED_EQUIPMENT_TAGS (422 if unknown tag) 4. If is_default=true, CRUD layer clears any other default for this user first 5. Backend creates row, returns TrainingEnvironmentResponse (201) 6. Frontend appends it to local list / refetches the page's environment list

Generating a Program (placeholder)

1. User clicks "Generate Program" on an environment card 2. ProgramCreationForm collects days/week, days, duration, start date, focus areas, weight unit + increments, progression style 3. Frontend POST /api/v1/programs { environment_id, ...preferences } 4. Backend looks up environment_id scoped to current_user.id → 404 TrainingEnvironmentNotFoundError if missing/foreign 5. Backend raises HTTPException(501, "Program generation is not yet implemented.") 6. Frontend catches the 501 specifically, shows a "coming soon" success-toned message

Deployment Considerations

Database Migrations

docker-compose exec backend alembic upgrade head

API Versioning

Both new routers live under /api/v1/, consistent with existing endpoints.

Backward Compatibility

Any existing client still sending equipment in a POST /users/profile body will have that field silently ignored by Pydantic (extra fields aren't rejected by default) rather than erroring — but the profile response will no longer contain it.

Rollback Plan

  1. Remove the /environments route and header nav entry (frontend-only revert)
  2. alembic downgrade -1 to restore UserProfile.equipment (environment data is lost — acceptable pre-production)
  3. Redeploy the previous backend build

Troubleshooting

❌ 422 on create/update: "unknown equipment tag"

The tag sent isn't in ALLOWED_EQUIPMENT_TAGS. Frontend and backend allow-lists must be kept in sync manually (frontend/src/types/trainingEnvironment.ts vs. backend/app/core/constants.py) — there's no shared source of truth yet at this scale.

❌ 404 on an environment I just created

get_training_environment scopes every lookup by user_id — this is the ownership check, not a bug. Confirm you're authenticated as the same user who created it.

❌ Two environments both show "Default"

Shouldn't happen — _clear_existing_default runs inside the same request as the create/update that sets a new default. If seen, check for a direct DB write that bypassed the CRUD layer.

❌ An environment added during onboarding doesn't appear to save

This is almost always the wizard's Next button being clicked instead of the form's own "Save Environment" button — see Onboarding Flow Technical Reference for the fix (Back/Next are disabled while the inline form is open).
