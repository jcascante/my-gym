Overview

MyGym's production infrastructure on AWS is defined as code with Terraform and deployed automatically by a GitHub Actions workflow on every push to main. The backend runs on ECS Fargate behind an Application Load Balancer, PostgreSQL runs on RDS, and the frontend is a static build served from S3 through CloudFront. Authentication between GitHub Actions and AWS uses OIDC federation — there are no long-lived AWS access keys stored in GitHub.

Key Components:

  • Terraform-managed ECS Fargate service, ALB, and RDS PostgreSQL instance
  • Terraform-managed IAM roles, including a GitHub OIDC trust relationship
  • Separate bootstrap/ Terraform config for account-level resources (ECR repos, OIDC provider, CI/CD role) that must exist before the application stack can deploy
  • GitHub Actions workflow (.github/workflows/deploy.yml) with independent backend and frontend jobs
  • One-off ECS task for running Alembic migrations before rolling the service
  • Remote Terraform state in S3 with DynamoDB locking
Note: Terraform apply is intentionally not run by CI. Infrastructure changes are applied manually and reviewed via terraform plan — see Deployment Considerations.

Terraform vs. Reused Resources

Not everything referenced by Terraform is created by it. The application config (terraform/main.tf) intentionally treats some existing infrastructure as read-only data sources rather than managing it, to avoid Terraform taking ownership of resources that were provisioned outside this codebase (e.g. the default VPC, an already-issued ACM certificate, and the existing frontend bucket/CloudFront distribution).

Resource Managed by Terraform? Notes
Default VPC & public subnets ❌ Reused (data) data.aws_vpc.main, data.aws_subnets.public
ECR repository ✅ Created in bootstrap/ Referenced as data.aws_ecr_repository.backend from the application stack
Frontend S3 bucket & CloudFront distribution ❌ Reused (data) app.costabirra.com, distribution E15DPRRJ6MVP5X

RDS PostgreSQL instance ✅ Created modules/rds

Terraform Modules

Terraform is split into two independent state files/configs:

  • terraform/bootstrap/ — account-level resources that must exist before the application stack can be applied (ECR repos, GitHub OIDC provider, CI/CD IAM role). State key: gym-app/bootstrap/terraform.tfstate.
  • terraform/ (application) — the ECS/RDS/IAM/networking stack for the running app. State key: gym-app/application/terraform.tfstate.

Both back onto the same S3 bucket (691650344087-terraform-state) and DynamoDB lock table (691650344087-terraform-locks), but as separate state files so bootstrap and application changes can't collide or be applied together by accident.

bootstrap/modules/ecr

Path: terraform/bootstrap/modules/ecr/

Creates one ECR repository per entry in var.ecr_repositories (named {project_name}-{repository}), with:

  • Vulnerability scan on push (image_scanning_configuration.scan_on_push = true)
  • AES256 encryption at rest
  • A lifecycle policy that expires images beyond the most recent 10, keeping registry storage bounded

bootstrap/modules/iam

Path: terraform/bootstrap/modules/iam/

Creates the trust relationship that lets GitHub Actions authenticate to AWS without static credentials:

  • OIDC provider (aws_iam_openid_connect_provider.github) trusting token.actions.githubusercontent.com
  • IAM role github-actions-deploy, assumable only via sts:AssumeRoleWithWebIdentity where the OIDC subject claim matches repo:jcascante/my-gym:ref:refs/heads/main — i.e. only workflow runs on main in this exact repo can assume it
  • Three scoped inline policies attached to that role: github-actions-ecr (push images), github-actions-ecs (update the service, run the migration task, and iam:PassRole for the two ECS runtime roles), and github-actions-frontend (S3 sync + CloudFront invalidation)
Note: aws_iam_openid_connect_provider hard-errors the entire plan on a 404 if a provider already exists for that URL — a data lookup can't conditionally skip creation, since AWS returns NoSuchEntity rather than an empty result. If an OIDC provider already exists in the account, use terraform import instead of re-running detection logic. See the comment in bootstrap/modules/iam/main.tf.

modules/iam

Path: terraform/modules/iam/

Creates the two runtime roles ECS Fargate tasks assume — kept separate from the bootstrap CI/CD role above:

  • ecs_execution — used by the ECS agent itself to pull the container image and ship logs; attaches AWS-managed AmazonECSTaskExecutionRolePolicy
  • ecs_task — the application's own runtime identity; currently grants CloudWatchLogsFullAccess only

modules/backend_ecs

Path: terraform/modules/backend_ecs/

The core compute module — provisions everything needed to run the FastAPI backend as a load-balanced, auto-scaling Fargate service:

  • Security groups: ALB accepts 80/443 from the internet; ECS tasks only accept traffic from the ALB's security group on container_port
  • ALB: HTTP listener 301-redirects to HTTPS when an ACM cert is configured; HTTPS listener uses ELBSecurityPolicy-TLS13-1-2-2021-06; supports additional SAN certificates via aws_lb_listener_certificate
  • ECS cluster with Container Insights enabled, plus a CloudWatch log group (/ecs/{project}-{env}, 30-day retention)
  • Task definition: Fargate, awsvpc networking, container health check via curl localhost:{port}/health, environment variables injected from a map (see Secrets & Variables)
  • ECS service: deployment circuit breaker with automatic rollback on failed deployments; lifecycle { ignore_changes = [desired_count] } so Terraform doesn't fight the autoscaler's decisions
  • Autoscaling: target-tracking policy on ECSServiceAverageCPUUtilization at 70%, between min_tasks and max_tasks, 300s scale-in / 60s scale-out cooldowns

modules/rds

Path: terraform/modules/rds/

Provisions the production PostgreSQL database:

  • PostgreSQL 17.10 on db.t4g.micro
  • Security group only allows inbound 5432 from the ECS tasks' security group (var.ecs_security_group_id) — the database is not reachable from outside the app's own network path
  • publicly_accessible = false, storage encrypted, automated backups (configurable retention), a defined backup/maintenance window
  • skip_final_snapshot = false — a final snapshot is always taken if the instance is ever destroyed
Note: Because RDS isn't publicly accessible, nothing outside the ECS tasks' security group — including CI runners — can reach it directly. This is why the database migration step (below) runs as an ECS task rather than as a direct alembic invocation from the GitHub Actions runner.

GitHub Actions Workflow

File: .github/workflows/deploy.yml

Triggers on every push to main, or manually via workflow_dispatch. Two jobs run in parallel — backend and frontend — since they deploy to unrelated resources (ECS vs. S3/CloudFront) and don't need to be sequenced.

permissions: id-token: write # required for OIDC — no AWS_ACCESS_KEY_ID/SECRET stored in GitHub contents: read

backend job

  1. Checkout code
  2. aws-actions/configure-aws-credentials@v4 assumes secrets.AWS_DEPLOY_ROLE_ARN via OIDC — no static keys
  3. aws-actions/amazon-ecr-login@v2 logs Docker in to ECR
  4. Build the backend image, tag it both {git-sha} and latest, push both tags
  5. Run the one-off migration task — see Migration Step below
  6. aws ecs update-service --force-new-deployment — rolls the service onto the new image

frontend job

  1. Checkout code
  2. Setup Node.js 22, npm ci in frontend/
  3. npm run build with VITE_API_URL=https://api.costabirra.com baked into the static build at build time
  4. Assume the same OIDC deploy role
  5. aws s3 sync dist/ s3://{FRONTEND_BUCKET} --delete — mirrors the build output, removing stale files
  6. aws cloudfront create-invalidation --paths '/*' — forces CloudFront to serve the new build immediately instead of stale cached assets
Note: Because VITE_API_URL is a Vite build-time env var, it's compiled into the static JS bundle — changing it requires a new frontend build and deploy, not just an environment variable change.

Database Migration Step

Since RDS only accepts connections from the ECS tasks' security group, migrations can't run directly from the GitHub Actions runner. Instead, the backend job launches a short-lived ECS task that reuses the running service's own network configuration and task definition, overriding only the container's command:

aws ecs run-task \ --cluster $ECS_CLUSTER_NAME \ --task-definition "$TASK_DEF" \ --launch-type FARGATE \ --network-configuration "{...same SG/subnets as the live service...}" \ --overrides '{"containerOverrides":[{"name":"backend","command":["uv","run","alembic","upgrade","head"]}]}'

The workflow then polls with aws ecs wait tasks-stopped, reads the container's exit code, and fails the job (exit 1) if the migration didn't exit 0 — a failed migration blocks update-service, so the service is never rolled onto a schema the running migration didn't produce.

Why this design: reusing the live service's networkConfiguration and task definition means the migration task automatically inherits the same security group RDS whitelists, without needing a separate bastion host or VPN just to run alembic upgrade head from CI.

Secrets & Variables

The workflow expects the following configured in the GitHub repository settings:

Name Kind Purpose
AWS_DEPLOY_ROLE_ARN Secret ARN of the github-actions-deploy IAM role created by bootstrap/modules/iam
ECS_CLUSTER_NAME Variable Terraform output backend_ecs_cluster
ECS_SERVICE_NAME Variable Terraform output backend_ecs_service
FRONTEND_BUCKET Variable Terraform output frontend_bucket
CLOUDFRONT_DISTRIBUTION_ID Variable Terraform output frontend_cloudfront_id

All five map directly to Terraform outputs in terraform/outputs.tf, so after any infrastructure change that affects these resources, re-check the outputs and update the corresponding GitHub repo variables/secrets if they changed.

Data Flow: Push to Live

  1. Developer action: Push (or merge) to main
  2. GitHub Actions trigger: Both backend and frontend jobs start in parallel
  3. OIDC handshake: Each job's runner requests a short-lived token from GitHub's OIDC provider and exchanges it for temporary AWS credentials by assuming github-actions-deploy
  4. Backend path: image built → pushed to ECR → migration task run and verified → ECS service force-deployed with circuit-breaker rollback protection
  5. Frontend path: static assets built with the production API URL → synced to S3 → CloudFront cache invalidated
  6. Result: new backend containers and new frontend assets are live, independently of each other

Testing & Validation

Terraform

cd terraform terraform init terraform validate terraform plan # review before ever applying terraform apply # manual, after review — see note below

Run the same for terraform/bootstrap/ when changing ECR repos or the CI/CD IAM role.

Workflow

Use workflow_dispatch to manually re-trigger a deploy from the GitHub Actions UI without needing a new commit — useful for retrying a failed frontend sync without rebuilding the backend, for example.

Deployment Considerations

Why Terraform apply isn't automated

Per the comment at the bottom of deploy.yml, infrastructure changes are deliberately kept out of the automated pipeline:

# Terraform deployments are managed locally via `terraform apply` # Infrastructure changes are infrequent and require manual review for safety. # To deploy infrastructure changes: # 1. Update Terraform files in terraform/ or terraform/bootstrap/ # 2. Run `terraform plan` locally to review changes # 3. Run `terraform apply` manually after approval

Application deploys (image pushes, service updates, static asset syncs) happen many times a day and are low-risk/reversible; infrastructure changes (networking, IAM trust, database instance class) are rare and higher-blast-radius, so they get a human review gate instead of a merge-triggered apply.

Rolling Deploys & Rollback

  • ECS deployment_circuit_breaker automatically rolls back a deployment that fails to stabilize (e.g. new tasks failing health checks)
  • lifecycle { ignore_changes = [desired_count] } on the ECS service prevents Terraform from undoing autoscaler-driven scale-out/in on the next apply
  • Frontend deploys are effectively instant rollback-capable: re-run the workflow against a previous commit to re-sync the old build and re-invalidate CloudFront

Secrets in Terraform Variables

db_password, secret_key, and db_username are marked sensitive = true in variables.tf and must be supplied via a local .tfvars file (gitignored) or environment variables — never committed to the repo.

Troubleshooting

❌ Deploy job fails at "Configure AWS credentials" with an OIDC trust error

Cause: The workflow is running on a branch/repo that doesn't match the trust policy's StringLike condition (repo:jcascante/my-gym:ref:refs/heads/main).

Fix: Confirm the push is to main in this exact repository. A fork or a different branch will be rejected by design.

❌ Migration task fails or times out

Cause: Bad migration SQL, or the task can't reach RDS.

Fix: Check the CloudWatch log group /ecs/{project}-{env} for the migration task's output. If it's a connectivity issue, confirm the task used the service's current networkConfiguration (the workflow re-fetches this each run) and that the RDS security group still allows the ECS tasks' security group.

❌ "AccessDenied" on ecs:RunTask or ecs:UpdateService

Cause: The github-actions-ecs inline policy scopes actions to specific ARNs (gym-app-production-cluster, gym-app-production-backend). A renamed cluster/service or new environment name won't match.

Fix: Update the ARNs in bootstrap/modules/iam/main.tf to match, then re-apply the bootstrap stack.

❌ Frontend changes not visible after deploy

Cause: CloudFront cache invalidation is asynchronous — it can take a few minutes to propagate to all edge locations.

Fix: Wait a few minutes, or check invalidation status with aws cloudfront get-invalidation. Confirm the S3 sync step actually ran (check the job logs) before assuming it's a caching delay.

❌ terraform apply wants to destroy/recreate the OIDC provider or an existing resource

Cause: A resource referenced with data was accidentally converted to a managed resource, or vice versa, or the resource was created manually outside Terraform.

Fix: Never accept a plan that destroys shared production infrastructure without understanding why. Use terraform import to bring an out-of-band resource under management instead of letting Terraform recreate it.
